## Installation

Copy the component from `packages/ui/src/components/animated/morphing-dialog/` into your project.

```bash
bun add motion clsx tailwind-merge
```

## Usage

```tsx
import { Button, MorphingDialog } from '@/components/animated/morphing-dialog';

export function Example() {
  return (
    <MorphingDialog>
      <MorphingDialog.Trigger>
        <Button variant="outline">Settings</Button>
      </MorphingDialog.Trigger>

      <MorphingDialog.Content>
        <MorphingDialog.Title>Preferences</MorphingDialog.Title>
        <MorphingDialog.Description>
          Changes apply immediately to this workspace.
        </MorphingDialog.Description>
        {/* ...panel body... */}
      </MorphingDialog.Content>
    </MorphingDialog>
  );
}
```

`MorphingDialog.Trigger` wraps *any* element (a button, an avatar, a card) and shares a
`layoutId` with `MorphingDialog.Content`, so the trigger's shape morphs directly into the
dialog panel instead of the panel simply fading in on top of it.

For controlled usage, pass `isOpen` and `onOpenChange`:

```tsx
const [isOpen, setIsOpen] = useState(false);

<MorphingDialog isOpen={isOpen} onOpenChange={setIsOpen}>
  <MorphingDialog.Trigger>{/* ... */}</MorphingDialog.Trigger>
  <MorphingDialog.Content>{/* ... */}</MorphingDialog.Content>
</MorphingDialog>;
```

### Text that morphs too

`MorphingDialog.Title` (and `.Description`) can be placed *inside* the trigger's content as
well as inside `Content` — both instances share a `layoutId`, so if your collapsed trigger
already shows a title (e.g. a note card), it morphs its position/size directly into the
dialog's heading instead of cross-fading:

```tsx
<MorphingDialog.Trigger>
  <MorphingDialog.Title className="text-sm">Launch checklist</MorphingDialog.Title>
  <p className="text-xs text-muted-foreground">Finalize pricing page...</p>
</MorphingDialog.Trigger>
<MorphingDialog.Content>
  <MorphingDialog.Title>Launch checklist</MorphingDialog.Title>
  {/* editor body */}
</MorphingDialog.Content>
```

Only the `Title`/`Description` rendered inside `Content` owns the `id` used for
`aria-labelledby`/`aria-describedby` — the trigger-side copy is purely visual so no
duplicate DOM ids are ever produced.

## Accessibility

- The trigger is a real interactive element — `role="button"`, `tabIndex={0}`,
  `aria-haspopup="dialog"`, `aria-expanded`, activatable with **Enter** or **Space**, with a
  visible `focus-visible` ring.
- The panel is `role="dialog"` `aria-modal="true"`, with `aria-labelledby`/`aria-describedby`
  wired automatically once a `MorphingDialog.Title`/`.Description` mounts inside `Content`.
- **Escape** closes the dialog, clicking the overlay closes it, and the built-in close button
  (or any `MorphingDialog.Close`) closes it.
- Body scroll is locked (and restored to its prior value) while the dialog is open.
- Focus moves to the close button (or the panel itself when `showCloseButton={false}`) on
  open, and returns to the trigger on close — the trigger stays mounted (just made
  `invisible`) while open so focus restoration is instant and the shared-layout morph has a
  stable origin to animate back to.
- All motion respects `prefers-reduced-motion`: the shared-layout morph is replaced with a
  plain crossfade, and the content-reveal delay is skipped.
